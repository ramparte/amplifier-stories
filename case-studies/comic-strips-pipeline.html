Architecture

The Thin Bundle Pattern

bundle.md is only 70 lines — a composition manifest, not a monolith. Three bundles compose to form the full pipeline:

amplifier-bundle-stories

Narrative creation capability. The storyboard-writer delegates into this bundle for story arc construction and pacing.

amplifier-bundle-browser-tester

QA screenshot capability. The compositor delegates to the visual-documenter for validating final HTML output.

comic-strips:behaviors

The actual pipeline — 6 agents, 3 tool modules, recipes. This is the behavior file that enumerates the wiring.

Key Insight

Cross-bundle delegation means each agent carries only the context it needs. The stories bundle handles narrative. Browser-tester handles QA. Comic-strips handles visual creation. No single agent carries all three concerns.

Routing

Leveraging the Routing Matrix

Each agent declares exactly the model capabilities it needs via model_role arrays. The routing matrix selects the best available model for each role.

Agent Model Roles Why
style-curator creative general Pure creative text work — style selection and guide generation
storyboard-writer creative general Narrative structure, panel breakdowns, dialogue
character-designer creative general vision Creates and reviews character reference sheets
panel-artist image-gen vision creative general Full quad — generate, review, prompt-craft, fallback
cover-artist image-gen vision creative general Same full quad — cover composition and typography
strip-compositor vision creative general Analyzes images for overlay placement — no generation
Design Principle

image-gen only on agents that actually generate images. vision on all agents that inspect images. Swap providers, add new models — agents don't change.

Context Pressure

Recipes, URIs, and the Binary Data Problem

The Problem

A 10-panel comic with characters = 50+ MB of image data. In v7.3.0, {{research_data}} was carrying 500K+ tokens through every recipe step. By panel 4, context hit 1.9M tokens.

The comic:// Protocol

# Project-scoped (shared across issues) comic://project/characters/name # Issue-scoped (per-issue assets) comic://project/issues/issue-001/panels/panel_01 # Total recipe context: ~1 KB URIs + storyboard JSON + metadata strings

3-Stage Architecture

Stage 1 — Text Only (~$0.50)

Research → style → storyboard → validate layouts

🛡 APPROVAL GATE

Stage 2 — Characters (shared)

Character design with cross-project reuse

Stage 3 — Per-Issue Art

Sub-recipe per issue: panels + cover in parallel

Binary data is fully encapsulated in the comic_create tool. Agents never see file paths, base64 blobs, or raw bytes. Sub-recipes (issue-art.yaml, issue-retry.yaml) enable surgical recovery without full pipeline re-runs.

Style System

How Agents Discover and Learn New Styles

🔍

Dynamic Style Discovery

The style-curator reads context/styles/ directory listing at runtime — no hardcoded style names. Adding a new style = dropping a .md file. Zero code changes.

📐

Canonical Sections

Every style pack includes: Image Prompt Template, Color Palette, Panel Conventions, Panel Shapes, Text Treatment, Character Rendering, AmpliVerse Branding.

🧬

Character Rendering Coherence

The Character Rendering section is quoted verbatim into every character-designer prompt as a "STYLE COHESION" directive, ensuring all characters share the same visual DNA.

🔄

Cross-Project Reuse

Characters designed in project A can be reused in project B if the style matches, with a needs_redesign flag for style-mismatch adaptation. Skills load on demand, keeping base agent context lean.

Extensibility Scorecard

New style: drop a .md file  •  New model: add a ModelEntry dataclass  •  Custom style: user description generates a complete guide following the same template. Zero agent code changes. Zero recipe changes. Zero bundle updates.

Lessons Learned

Road Bumps Along the Way

Shadow-Operator's 43-Minute Container Gauntlet

First shadow test of the comic pipeline. Container refused to build — dependencies tangled, config paths missing inside the sealed environment. Second attempt: another collapse. Third try: switched to general-purpose container, manually configured from scratch.

"Prove it in isolation first. Shadow-operator's gauntlet was not wasted time. It was the foundation that made everything after it possible."

The Caching Productivity Trap

The install-state.json vs cache/ mismatch bug — reset cleared the cache but not the state file, creating a "phantom installed" condition. ModuleActivator thought modules were installed but the code was gone.

Lesson: State tracking files should live in cache/ if they track cached data.

Forcing Functions

Shadow testing failed TWICE on premature fixes — forced the team to actually understand root causes (session creation flow bypassing AppModuleResolver) before declaring success.

Also: rm -rf ~/.amplifier/cache/ breaks the CLI entirely — site-packages contain links to cache files. Prescribed: amplifier reset --remove cache,registry -y

Takeaways

What We Learned

1

Composition over monoliths

Thin bundle + behavior file + cross-bundle delegation = each agent carries only what it needs.

2

Route capabilities, not providers

The routing matrix (and its image-gen sibling) let you swap models without touching agent code.

3

URIs, not bytes

The comic:// protocol keeps recipe context at ~1 KB while managing 50+ MB of images.

4

Extensibility through convention

New styles are .md files. New models are ModelEntry entries. No code changes required.

5

Shadow testing is painful but essential

43 minutes of container gauntlet prevented weeks of "works on my machine" bugs.

6

Cache coherence matters

State tracking must co-locate with the data it tracks — or you get phantom-installed conditions.
