The Origin

Powerful data.
Completely invisible.

Every Amplifier session generates a rich log of events — tool calls, agent delegations, responses, thinking blocks, errors. All stored as JSONL files. None of it visible to the developer.

~/.amplifier/projects/ -Users-me-my-project/ sessions/ a3f2c1e8-... metadata.json events.jsonl ← megabytes of rich data b7d4e9f1-... metadata.json events.jsonl

The idea was simple: a fast, lightweight web viewer that runs locally, auto-detects the current project, opens a browser, and lets you explore sessions instantly.

How It's Built

Lean by design

Python because Amplifier users already have it. Flask + HTMX for server-side rendering — multi-MB event files never leave the server. Pydantic for type-safe data. Two runtime dependencies total. One command to install.

Events are indexed in memory for O(1) filtering. List pages use binary line counting and tail-reads — they never parse the full file. Detail pages parse once, then cache the rendered HTML. A semantic color system — green for success, amber for attention, blue for interaction, red for errors — lets you scan sessions at a glance.

# Or try instantly, no install

The Widening Lens

From one session
to everything

The viewer was originally built for a single purpose: show me what happened in this session. But the scope kept expanding naturally — and each expansion unlocked something more valuable than the last.

1
One Session Built
"What happened in this conversation?" — Open a session, see every event, every tool call, every response. The original problem, solved.
2
One Project Built
"What's the full history of this project?" — All sessions for the current project, chronologically ordered, with parent/child agent hierarchy. Your project's complete development story.
3
All Projects Built
"What am I working on across everything?" — A bird's-eye dashboard of every project you've used Amplifier in. How many sessions, when you last worked on it, the full scope of your AI-assisted development.
4
All Machines Vision
"Show me everything, everywhere." — Your complete development history across every machine, synced and searchable. The natural next step.

The first three levels all work today. And the third one — the all-projects view — was the surprise. It turned the viewer from a debugging tool into a development dashboard.

The Discovery

Development history
lives outside the code

While building this project, a fundamental tension surfaced.

Today

Development sessions — the how and why behind every decision — are stored in ~/.amplifier/. They live on one machine. When you clone a repo, you get the code but not the process that created it.

What If

An option to store session data inside the project — .amplifier/sessions/ — would make development history portable, shareable, and version-controlled. Clone a repo, get the full AI-assisted story behind every feature.

Imagine cloning a repo and getting not just the code,
but the full development conversation behind every feature.
Vision — Continuity

Your history,
everywhere

Today, your development history is trapped on one machine. But development doesn't happen in one place.

Morning — Laptop at the coffee shop
You start debugging a tricky authentication issue. Three sessions deep, you've identified the root cause and have a fix in progress.
Afternoon — Desktop at home
You sit down and pick up exactly where you left off. Every session, every conversation, every decision — already there.
Evening — Phone on the couch
A teammate asks about the fix. You pull up the session on your phone, share the link, and they see the full development journey.

No manual export. No "sync my sessions" step. Start a session on one machine, continue on another — your entire development history, accessible from anywhere.

Vision — Intelligence

Intelligence
across every project

Each Amplifier session is a structured record of developer-AI collaboration. Across hundreds of sessions, across dozens of projects, patterns emerge.

Universal Search
Search across all your projects at once. "Where did I solve that Redis connection issue?" — find it in seconds, even if it was three months ago in a different project.
Development Analytics
How long do features take? What types of bugs recur? Which tools get used most? Where do sessions stall? Data-driven insights on how you actually develop.
Pattern Recognition
"You solved a similar problem in project X three months ago." — ML on top of your development history, surfacing relevant past work automatically.
Development Replay
Revisit how a complex feature was actually built. Not the git log — the full thinking process, dead ends included, as they happened.
Vision — Collaboration

Share the
thinking, not
just the code

Team Sharing
Share sessions with teammates — not just a diff, but the full conversation and reasoning. See how a colleague solved a problem, learn from their AI-assisted approach.
Project Handover
Hand over a project with full development context. The new maintainer doesn't just get code — they get the why behind every architectural decision.
Open Source History
Open source projects could include development sessions. Clone the repo, get the code AND the full AI-assisted development story behind every feature.
A community of developers sharing not just what they built,
but how they think about building.

Junior developers learn by watching how senior developers work with AI. Senior developers discover techniques from each other's sessions. Everyone develops faster because development knowledge becomes a shared, explorable resource — not something locked in one person's head.

Reimagining PRs

Sessions as the
unit of work

Pull Requests Today
A diff — what changed
A description — what the author chose to explain
Comments — post-hoc review after the fact
The process behind the code is lost forever.
PRs Built on Sessions
Each bug fix or feature = one Amplifier session
PRs become a list of sessions showing how the developer actually worked
Ongoing PRs = active sessions that can be handed to another developer mid-stream
Code review = review the code AND understand the full development journey

A reviewer sees not just that a function was refactored, but the three approaches the developer considered, why two were rejected, and the conversation that led to the final design. Dead ends become learning opportunities. The reasoning behind the code becomes a permanent, reviewable artifact.
