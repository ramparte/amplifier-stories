Reality Check Technical Walkthrough

How amplifier-bundle-reality-check turns user intent into evidence-backed pass/fail.

Author David Koleczek

Reality Check sits on top.

It leverages everything below it — the DTU for an isolated environment, browser-tester and terminal-tester for in-environment interaction, and amplifier-gitea when a repo needs to be served — and adds the missing piece on top: did the software actually do what the user wanted?

Reality Check is the top layer of the cake.
Pipeline

Five steps, one recipe.

The reality-check-pipeline recipe orchestrates everything. Intent and DTU are independent — conceptually parallel — and validators only run on tests they actually own.

Intent Analyzerconversation → tests.yaml
DTU Profile Builderprofile → running env
Validatorsterminal · browser · generic
Report Agentresults → report.raw.yaml
CLI Renderreport.yaml + report.html

Two convergence loops: one wrapping the intent analyzer, one wrapping the report agent. The CLI — not the LLM — decides if output is valid.

Step 1

Intent Analyzer.

Produces a single tests.yaml file of structured acceptance tests — the source of truth for everything downstream. By default it derives them from the inputs below, but it’s flexible: you can also hand it something that already looks like acceptance tests and it will leverage that directly.

Default inputs

  • Full conversation history
  • Spec or README (optional)
  • Repo paths (optional)
  • Output path

Or skip these and pass in your own draft acceptance tests — the analyzer will validate, fill gaps, and normalize.

Output schema

  • summary, software_type
  • entry_points — URLs, commands
  • tests[] — id, type, steps[]
  • assumptions[] — what it inferred
Step 1 · Example output

What a real tests.yaml looks like.

Synthesized for a URL-shortener web service — one test per validator type.

summary: "FastAPI URL shortener with a web UI, JSON API, and admin CLI." software_type: web_app entry_points: - { type: url, value: "http://localhost:8080", label: "Web UI" } - { type: command, value: "shortener admin", label: "Admin CLI" } tests: - id: a3f12c8b # 8-char hex, CLI-injected description: "Web UI shortens a URL and shows the short link" type: browser steps: - action: "Navigate to http://localhost:8080" expect: "Page renders with a 'Long URL' input and 'Shorten' button" - action: "Enter https://example.com and click Shorten" expect: "Result panel shows http://localhost:8080/<6-char-code>" - id: 9d4a210e description: "API resolves a short code via 302 redirect" type: other steps: - action: "curl -sX POST localhost:8080/api/shorten -d '{\"url\":\"https://example.com\"}'" expect: "HTTP 200, body is JSON with non-empty short_code" - action: "curl -sI --max-redirs 0 localhost:8080/<short_code>" expect: "HTTP 302 with Location: https://example.com" - id: 7e80f3a1 description: "Admin CLI lists shortened URLs as JSON" type: cli steps: - action: "shortener admin list --format json" expect: "Exit 0, stdout is a JSON array containing the short_code from earlier tests"
Step 1 · Convergence

The CLI is the convergence judge.

The agent writes tests.yaml. A bash step in the recipe runs amplifier-reality-check validate-acceptance-tests. If it exits non-zero, raw pydantic errors are fed back into the next attempt. Capped at 3 iterations.

intent-analyzer
writes tests.yaml
→
validate-acceptance-tests
pydantic check
→
pass → continue
fail → retry with errors
# reality-check-pipeline.yaml — intent loop while_condition: "{{validated}} != true" max_while_iterations: 3 break_when: "{{validated}} == true" update_context: validated: "{{iteration_result.validation_result.validated}}" previous_errors: "{{iteration_result.validation_result.errors}}" # fed back in

Side effect: validate-acceptance-tests auto-injects an 8-char hex id into any test missing one, then rewrites the file in place.

Step 2 · DTU Profile Builder

Stand up the universe.

Delegated to digital-twin-universe:dtu-profile-builder. Independent of the intent analyzer — conceptually parallel.

What it does

  • Explores the project and infers ports, deps, runtime needs
  • Generates a DTU profile YAML
  • Launches the environment with amplifier-digital-twin launch
  • Verifies readiness, hands back URLs / exec commands

What downstream gets

  • A DTU instance ID (e.g. dtu-a1b2c3d4)
  • Forwarded URLs for browser tests (e.g. http://localhost:8080)
  • An exec channel for terminal tests — amplifier-digital-twin exec <id>
  • The deployed software, ready as a real user would see it

Stays running after the pipeline finishes — the user can poke at the deployed software and inspect failures interactively.

Step 3 · Validators

Three validators, one rule: only run what you own.

Each validator filters tests.yaml by type and exits immediately if no tests match. They run sequentially in the recipe.

terminal-tester

Owns: type: cli

Spawns CLI/TUI apps inside the DTU via the terminal_inspector PTY bridge. Sends keystrokes, captures screen state, verifies output as a real user would.

browser-tester

Owns: type: browser

Drives a real browser against the DTU's forwarded URL via the vision-capable browser-operator sub-agent. Navigates, fills, clicks, screenshots.

generic-tester

Owns: type: other

Catch-all. Runs bash inside the DTU — HTTP probes, file checks, library imports, anything reducible to “run a command, check output.”

Step 3 · Validator outputs

Each returns the same shape.

A markdown results table. The Report Agent ingests these blocks verbatim — it doesn't rerun anything.

Step 4 · Report Agent

Aggregate, don't re-judge.

The agent reads tests.yaml + the three validator result blocks and writes a single slim file: report.raw.yaml. Top-level extra=forbid — only results: is allowed.

# report.raw.yaml — one entry per acceptance test results: - id: a3f12c8b # must match an acceptance-test id status: pass # pass | fail (no skip, no error) evidence: "Web UI rendered short link http://localhost:8080/Xk7mNq" screenshots: ["02-a3f12c8b-result-displayed.png"] - id: 9d4a210e status: pass evidence: "curl -I returned 302, Location: https://example.com" - id: 7e80f3a1 status: pass evidence: "`shortener admin list --format json` exit 0, output contained Xk7mNq" screenshots: ["03-7e80f3a1-admin-list.png"]

Why slim? Statistics, descriptions, and bucket assignments are computed deterministically by the CLI — not asked of the LLM.

Step 4 · Convergence + render

CLI validates, then renders.

Same convergence pattern as the intent analyzer. After the agent settles, the CLI joins the raw report against the acceptance tests and emits two derived artifacts.

report agent
writes report.raw.yaml
→
validate-report
schema + id match
→
renders
report.yaml + report.html

report.yaml

Canonical, expanded. Joins on id, buckets into passed / failures / missing, computes statistics (total, pass_rate, …), copies through assumptions.

report.html

Self-contained visual artifact for end users — auto-generated. Optional --dtu-details-file renders an “Environment Access” footer so a reader can connect to the DTU and reproduce.

Loop guarantees: capped at 3 iterations, errors fed back to the agent on retry, recipe fails loudly if convergence is never reached.

Intent → tests.
Tests → evidence.
Evidence → report.

Every step has a deterministic checkpoint — the CLI — sitting between LLM output and the next stage. Convergence is a recipe property, not a model promise.
