Part 1

Why Build the Digital Twin Universe?

Closing the gap between "tests pass" and "it actually works."

Why · Evidence-Based Testing

The original motivation.

DTU was built to provide the infrastructure for evidence-based testing. When a software factory produces a unit of work — an app, an Amplifier component, anything — one critical question remains: does it actually work when a user goes to use it?

The gap

We had nothing that could seamlessly set up an environment an agent could use to validate in a way that mirrors the real world as closely as possible. This is where the "digital twin" part of the name comes from.

The fix

A clean, non-dev environment, stood up on demand, that looks like a fresh user machine. The resolver (or developer) pushes changes in from the outside and tests against it — the DTU can't see the dev environment, so results are evidence, not echoes.

From "tests pass on my machine" to "this actually mirrors what would happen in production."
Why · What Came Before

Shadow-env got us partway there.

For Amplifier development, we had tools like the shadow environment. But it had limitations that made it difficult to correctly mirror what would happen if code was pushed to GitHub — our notion of production.

Amplifier-specific

Built for Amplifier ecosystem testing. Couldn't stand up broader systems that interact together — mock services, web UIs, full user-facing stacks.

CLI-only in practice

OK for CLI-based testing, or agent-driven for anything further. Not useful for GUI-level experiences or beyond.

Leaky isolation

Testing could impact the dev environment. The fallback was "test in prod," hack around it, or go to a literally clean machine.

Narrow scope

Our use cases go beyond testing within the Amplifier ecosystem — arbitrary software, full deployments, agent-driven validation loops.

Part 2

The Universe

Why universe — and how Incus makes it possible.

The Universe · Infrastructure

A digital twin universe, not just a container.

We're building increasingly complex things autonomously. The software interacts with other services, external services, and often has a whole host of dependencies. The digital twin universe must be able to provide the infrastructure to manage all of that complexity.

Incus system containers

A system container behaves much more like a lightweight VM: it boots an init system (systemd), runs multiple processes, has persistent state. You SSH in, install packages, and manage it like a regular Linux server — without the overhead of hardware virtualization.

Full VMs too

Incus can manage actual KVM virtual machines through the same interface, so containers and VMs live side by side. This is what lets us create a universe that closely matches the end-user's experience.

The Universe · Mocks

Mock services, when the real thing is tedious or infeasible.

We're building out an ecosystem of mock services so agents and developers can test against realistic surfaces without touching production APIs, rate limits, or manual setup.

Mock Slack

A basic mock Slack service that apps can point at like the real API. Handy for amplifier-hive-slack-style scenarios with zero source-code changes.

amplifier-bundle-gitea

Ephemeral Gitea instances that mock GitHub. Mirror repos in, work freely, optionally promote back as PRs. Our standard stand-in for GitHub inside DTUs.

Deployment choice: both can run inside a DTU or alongside it on the host, depending on your use case.

Part 3

The Universe Around the Digital Twin Universe

Use cases we learned about while building — and the bundles that emerged.

Around the DTU · Three Emerging Use Cases

Each use case got its own bundle or pattern.

Testing Amplifier itself

One pattern we saw was using DTU as a more robust means of testing Amplifier. So we created amplifier-tester — agents and context for using Gitea + DTU for testing within the Amplifier ecosystem.

Testing anything else

reality-check takes browser-tester, terminal-tester, and friends and makes them work out of the box with DTUs. It also has an intent analyzer — given conversations or specs, can we generate acceptance tests for validation automatically?

Dev environments

Since Incus containers behave like full systems, you can stand up many isolated dev environments in parallel — e.g., developing multiple Amplifier components without touching your system installation.

Reality-check flow: provide intent or acceptance tests → a DTU is stood up → appropriate tests run → a final report/analysis is returned. Completely isolated.

Part 4

FAQ

Questions that came up on the team.

FAQ · Profiles & Workflow

Common questions, short answers.

Are there standard DTU profiles, or does everyone roll their own?
Both, and we're still figuring out the right balance. Profiles live next to the thing they test — e.g., a repo can ship its own profile under .amplifier/digital-twin-universe/profiles/ for self-validation. Over time we expect common patterns (and a shared mock service catalog) to emerge and get standardized. For now, use the profile-builder agent; it is good at creating custom ones on demand fairly quickly and reliably.
How do I close the loop when reality-check finds issues?
This is still an open area. The current pattern is to feed the findings back into your dev session as context and iterate — either by hand or via something like brainstorm. We don't have a canonical fix-loop yet, so feedback here is welcome.
Does reality-check generate its own acceptance tests?
Yes. It can derive acceptance tests from your conversation history, spec, or project context. You can also pass in your own acceptance tests explicitly, and it will leverage any combination of the two.
FAQ · Scope & Positioning

More common questions.

Is this going to replace shadow-env?
Yes.
Is this integration testing? When should I reach for DTU or reality-check versus unit tests?
Roughly, yes — this is closer to integration/acceptance testing than unit testing. DTU gives you a clean environment that looks like a real user's machine, and reality-check runs acceptance tests on top of it. Use them after an implementation to verify it actually works end-to-end, not to replace your unit tests.
FAQ · Docker

Does this replace Docker?

Not necessarily. Incus and Docker both use Linux container technology, but they're designed for different purposes.

Docker — application containers

Each container runs a single process or service. Starts from an immutable image, ephemeral by design: throw it away and spin up a new one when something changes. Philosophy: "package one app with its dependencies and ship it."

Incus — system containers (and VMs)

Behaves like a lightweight VM: boots an init system, runs multiple processes, has persistent state. SSH in, install packages, manage it like a regular Linux server. Also natively manages KVM VMs through the same interface.

Lifecycle & scope

Docker: disposable, rebuilt from images. One app per container. Incus: long-lived, mutable, snapshotted. Whole OS userspace with many services.

Networking & storage

Incus has richer built-in primitives for bridges, ZFS/Btrfs/LVM pools, clustering, live migration — closer to a hypervisor manager like Proxmox.

Reach for Docker when packaging and deploying applications. Reach for Incus when you want lightweight Linux machines (or a mix of containers and VMs) to use as infrastructure.
