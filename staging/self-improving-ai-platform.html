Design Principles

Four pillars of AI-friendly design

📡

Simple HTTP

Every service is a plain REST API on its own port. No frameworks, no middleware chains. AI agents can discover and call any endpoint with a simple HTTP request.

📁

File-Tree Storage

Data lives as JSON files on disk — human-inspectable, git-diffable, AI-parseable. No database servers, no ORMs, no migration scripts.

📜

NL Specifications

The "Natural Language Spec" philosophy: write what you want in plain English, let AI help implement. Technology-agnostic design docs that anyone can read.

📦

JSON Contracts

Services talk via JSON over HTTP. Minimal upfront interface design — just enough structure for AI to understand, flexible enough to evolve fast.

The key realization: What makes services easy for AI to use also makes them easy for humans. Simple REST, file-based state, clear contracts — good design converges.

Git as Infrastructure

The database you already know

A storage abstraction library with swappable backends. The Git provider turns any repo into a versioned, syncable data store — every write is an atomic commit.

# StorageProvider protocol interface: read / write / delete / list / exists / sync data: bytes in, bytes out paths: forward-slash strings # Git backend behavior write() → auto-commit per operation sync() → pull --rebase, then push read() → plain file read from working tree # Data layout repos/{owner}/{repo}/issues/{n}/issue.json repos/{owner}/{repo}/counter.txt

Why Git?

⏳ Version History

Every state change is tracked. Revert any operation, inspect any point in time.

👀 Human-Inspectable

JSON files in a directory tree. cat, grep, diff — every Unix tool works.

📈 Natural Scaling

Solo → cross-device → small team → larger team. Just share repo access.

⚡ Zero Infrastructure

No database server. No connection strings. No migrations. Git is already everywhere.

Containerization

Containers as composition units

A container bundles a mix of services for a specific scenario. Today it's a single Dockerfile; tomorrow it's Docker Compose. Clone repos, run from source — no published packages or images needed.

FROM python:3.13-slim COPY --from=ghcr.io/astral-sh/uv /uv /usr/local/bin/ # Clone repos and run directly from source COPY gh-issues-local/ ./gh-issues-local/ RUN uv sync --frozen --no-dev ENV GH_ISSUES_LOCAL_DATA_DIR=/data EXPOSE 10100 CMD ["uv", "run", "gh-issues-local", "--production"]

Philosophy

📦 Source-First

Clone repos and run directly. No registry, no artifact store, no version pinning headaches.

⚡ Fast Builds

Python 3.13 + uv for reproducible, fast dependency resolution. Slim images, minimal layers.

🔐 Secure by Default

Services bind to localhost only. Network access is explicit opt-in. Token auth activates when bound externally.

📀 Volume Mounts

Data dir mounted as a volume. Persistent state survives container restarts. Dotfiles repos for env customization.
