Executors

Three executor types let Loom run anything — from shell scripts to full AI sessions

shell

Any shell command or script. The classic — now with live streaming output, history, and scheduling superpowers.

daily-backup.yaml
id: daily-backup trigger: type: cron schedule: "0 2 * * *" executor: type: shell command: rsync -av ~/projects/ /backup/
claude-code

Claude AI sessions as first-class jobs. Model selection, chaining, multi-step reasoning — all schedulable.

pr-review.yaml
id: review-pr trigger: type: connector entity: github.pr/myorg/repo executor: type: claude-code model: claude-opus-4-5 prompt: | Review this PR for quality, security, and test coverage.
amplifier

Full Amplifier AI sessions or .yaml recipe files. Deep bidirectional integration — Amplifier can create Loom jobs too.

inbox-processor.yaml
id: process-inbox trigger: type: watch path: ~/Inbox events: [create] executor: type: amplifier recipe: process-doc.yaml

Key point: The three executor types map to three eras of automation: shell is the Unix era, claude-code is the AI-native era, and amplifier is the multi-agent workflow era. Every executor gets the same scheduler infrastructure — streaming output, history, pause/resume, live UI.

If challenged: "What's the difference between claude-code and amplifier?" claude-code runs a single Claude session via claude -p "...". amplifier runs a full Amplifier session or a .yaml recipe with full tool access, skill loading, and multi-agent delegation. Amplifier is the more powerful executor; claude-code is lighter weight for simpler tasks.

Triggers

Five trigger types — and only one fires when the world actually changes

🕐 cron
Standard cron expressions — but with seconds precision. "0/30 * * * * *" — every 30 seconds
🔁 loop
Repeating interval — clean, readable, no cron syntax required. interval: 5m — fire every 5 minutes
🎯 once
Run once at a specific time, then auto-disable. No cleanup needed. at: "2025-12-31T23:59:00" — then disabled
👁 watch
OS-level filesystem events via inotify / FSEvents / kqueue. Zero polling. path: ~/Inbox, events: [create, modify]
⚡ connector
Fires only when an external service actually changes. Powered by the Mirror system. Entity-addressed. entity: github.pr/myorg/repo · amazon.product/B09V3KXJPB
💡

The connector trigger is what makes Loom categorically different from any other scheduler. Most schedulers poll on a timer and fire regardless. Loom's Mirror watches external services, diffs the results field-by-field, and fires jobs only when something genuinely changed. Zero wasted AI compute. Zero noise.

Key point: The first four triggers are excellent — they cover the full spectrum of time-based scheduling. But the connector trigger is in a different category entirely. It transforms Loom from a "run things on a schedule" tool into a "react intelligently to the world" platform. The Mirror system does the heavy lifting.

If challenged: "Can't you just poll and check if something changed in a shell script?" You can, but you're now managing state across runs, handling diff logic, and re-inventing the Mirror system manually for every job. Loom gives you that infrastructure with entity addressing, field-level diffs, and structured change events — once, for free, for any connected service.

Key point: The Mirror subsystem is Loom's secret weapon. It maintains a local shadow copy of external services, computes field-level SHA-256 diffs on every poll, and fires the connector job only when something in that diff is non-empty. Jobs get the full structured change event as environment variables — they don't need to re-fetch or parse anything.

If challenged: "What about rate limiting? What if I have 20 jobs watching the same GitHub repo?" The Mirror is the fetch layer — it polls once per configured interval and fans out to all waiting connector jobs. Multiple jobs watching the same entity do not multiply API calls. The mirror handles deduplication automatically.

Architecture

One Go binary, embedded React SPA, zero runtime dependencies — the whole thing ships in a single curl

Single Binary
The entire React SPA is compiled by Vite and go:embed-ed into the binary at build time. One file ships it all.
Zero Runtime Deps
No PostgreSQL, Redis, or Node process required. bbolt is a pure-Go embedded key-value store. curl | bash and you're running.
SSE Ring Buffer
Streaming output is broadcast via SSE with a ring buffer. Refresh the browser mid-run and you'll catch up on everything you missed.
WebSocket PTY Bridge
A full PTY bridge over WebSocket powers the embedded terminals. Real xterm.js sessions in the browser.

Key point: The architectural decision to embed everything in a single binary is deliberate. It makes distribution trivial, removes deployment ceremony entirely, and means there's no version mismatch between the frontend and backend. The binary you download is the complete, runnable product.

If challenged: "What happens to job history when the binary is upgraded?" bbolt persists to disk (typically ~/.loom/loom.db). History, mirror state, and job definitions survive binary upgrades. The auto-updater in the macOS tray handles upgrades without losing state.
