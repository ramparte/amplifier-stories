Part 1

Why Build the Digital Twin Universe?

Closing the gap between "tests pass" and "it actually works."

The original motivation.

DTU was built to provide the infrastructure for evidence-based testing — making sure that when a software factory produces a unit of work, it actually works when a user goes to use it.

The gap

Nothing that could seamlessly set up an environment an agent could validate against, in a way that mirrors the real world — hence the "digital twin" in the name.

The fix

An on-demand environment that looks like a fresh user machine. The agent pushes changes in from outside and tests against it — results are evidence, not echoes.

Shadow-env got us partway there.

Amplifier-specific

Focused on the Amplifier ecosystem. Our use cases go beyond that — arbitrary software, full deployments, agent-driven validation loops.

Didn't mirror production

Hard to correctly mirror what would happen if code was pushed to GitHub (aka production)

Part 2

The Universe

Why universe — and how Incus makes it possible.

A digital twin universe, not just a container.

We are building increasingly complex things autonomously — software that interacts with other services, external APIs, and often a whole host of dependencies. The DTU has to provide the infrastructure to manage that complexity.

Incus — system containers

Behaves like a lightweight VM: boots an init system, runs multiple processes, has persistent state. SSH in, install packages, manage it like a regular Linux server — without hardware virtualization overhead.

Why it matters

This is what lets us create a universe that closely matches the end user's experience.

Profiles define a universe, declaratively.

Declarative YAML files that define a complete DTU environment — services, networking, provisioning, and access ports — self-contained and launchable on their own.

Example · amplifier-chat: launches a browser-accessible chat UI with port 8410 forwarded, so you can open http://localhost:8410/chat/ immediately after launch.

What a profile looks like.

name: amplifier-chat base: image: ubuntu:24.04 access: # forward to localhost ports: - { host: 8410, container: 8410, label: "Chat UI" } passthrough: # real APIs via the boundary services: - { name: anthropic, key_env: ANTHROPIC_API_KEY } provision: # what to install & start setup_cmds: - apt-get install -y git curl - curl -LsSf https://astral.sh/uv/install.sh | sh - uv tool install "amplifier-chat[standalone] @ git+..." - # ...configure provider, start server on 8410 readiness: # when is it "ready"? - http: { url: http://localhost:8410/ready }

Profiles can travel with bundles.

Profiles can also be saved as templates or parameterized with environment variables — including them inside bundles as context or tests.

Useful for: providing sample data, templates for custom repos, or parameterizing ports — keeping self-validation next to the code it validates.

Mocks, when the real thing is tedious or infeasible.

We also are hoping to have an ecosystem of mock services that we can use when using the real thing is tedious or infeasible.

Mock Slack

A basic mock for Slack — see the community mocks list in the DTU docs.

amplifier-bundle-gitea

Lets us mock GitHub — ephemeral Gitea instances we can mirror repos into, work against, and optionally promote back as PRs.

Deployment choice: both of these can then be either managed inside of a DTU or on the side depending on your use case.

reality-check.

For testing arbitrary software (like what Amplifier Resolve produces). Wires up browser-tester and terminal-tester to work with DTUs out of the box, plus an intent analyzer that derives acceptance tests from conversations, specs, or project context.

Isolated flow: intent / acceptance tests → DTU is stood up → validators run → gap analysis report.

Trying what you or others build.

By sharing a profile, you can easily give someone a way to try something out, as they are self-contained.

Hand someone a profile → they launch it → they're running your thing in an environment that looks like a real user's machine, no matter what state their dev box is in.

Parallel development environments.

Since the underlying Incus containers that DTUs use are like entire systems, it makes it possible to create many isolated development environments in parallel.

For example: if you wanted to develop multiple Amplifier components without them modifying your system installation.

Part 4

FAQ

Questions that came up on the team.

Common questions, short answers.

Are there standard DTU profiles, or does everyone roll their own?
Both, and we're still figuring out the right balance. The convention is that profiles live next to the thing they test. For example, a repo can ship its own profile under .amplifier/digital-twin-universe/profiles/ for self-validation. Over time we expect common patterns (and a shared mock service catalog) to emerge and get standardized. For now, use the profile-builder agent; it is good at creating custom ones on demand fairly quickly and reliably.
How do I close the loop when reality-check finds issues?
This is still an open area. The current pattern is to feed the findings back into your dev session as context and iterate — either by hand or via something like brainstorm. We don't have a canonical fix-loop yet, so feedback here is welcome.
Does reality-check generate its own acceptance tests?
Yes. It can derive acceptance tests from your conversation history, spec, or project context. You can also pass in your own acceptance tests explicitly, and it will leverage any combination of the two.

More common questions.

Is this going to replace shadow-env?
Yes.
Is this integration testing? When should I reach for DTU or reality-check versus unit tests?
Roughly, yes — this is closer to integration/acceptance testing than unit testing. DTU gives you a clean environment that looks like a real user's machine, and reality-check runs acceptance tests on top of it. Use them after an implementation to verify it actually works end-to-end, not to replace your unit tests.

Does this replace Docker?

Not necessarily. Incus and Docker both use Linux container technology, but they're designed for different purposes.

Docker — application containers

Each container typically runs a single process or service, starts from an immutable image, and is meant to be ephemeral — you throw it away and spin up a new one when something changes. Philosophy: "package one app with its dependencies and ship it."

Incus — system containers (and VMs)

Behaves much more like a lightweight VM: boots an init system, runs multiple processes, has persistent state. SSH in, install packages, manage it like a regular Linux server. Also natively manages KVM VMs through the same interface.

Lifecycle & scope

Docker: disposable, rebuilt from images. One app per container. Incus: long-lived, mutable, snapshotted. Whole OS userspace with many services.

Networking & storage

Incus has richer built-in primitives for bridges, ZFS/Btrfs/LVM pools, clustering, live migration — closer to a hypervisor manager like Proxmox.

Reach for Docker when packaging and deploying applications. Reach for Incus when you want lightweight Linux machines (or a mix of containers and VMs) to use as infrastructure.
